## Domain-Driven Design

This project follows patterns from "Implementing Domain-Driven Design" (Vaughn Vernon).

Use the DDD skills to guide implementation and review:

- `/ddd-guide` — unsure which pattern to use? Start here
- `/ddd-ubiquitous-language` — aligning names with domain expert vocabulary
- `/ddd-bounded-context` — designing or reviewing system/service boundaries
- `/ddd-aggregate` — modeling objects that must stay consistent together
- `/ddd-entity-value-object` — deciding between entity and value object
- `/ddd-repository` — implementing persistence for aggregate roots
- `/ddd-factory` — complex object creation
- `/ddd-services` — placing domain vs application logic
- `/ddd-domain-event` — cross-aggregate communication via events
- `/ddd-event-sourcing` — storing state as an event log
- `/ddd-cqrs` — separating read and write models
- `/ddd-saga` — multi-step workflows across aggregates
- `/ddd-anti-corruption-layer` — protecting the domain from external models
- `/ddd-specification` — composable business rules and query criteria
- `/ddd-modules` — packaging that reflects the Ubiquitous Language
- `/ddd-context-map` — integration relationships between bounded contexts (Partnership, ACL, OHS, etc.)
